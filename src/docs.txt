#
Архитектура гексагональная (изнутри - наружу):
- доменные сущности (из DDD)
- доменные сервисы (обычно в них лежат методы, которые не понятно, в какой доменной сущности разместить)
- сервисы приложения (они же юз-кейсы)
- презентационный слой (интерфейсы реквест-респонс)
- инфраструктура (БД, Nest с контроллерами, другие фреймворки...)

#
Как это может выглядеть в структуре каталогов:
application (сервисы приложения)
config
domain
infrastructure
presentation
utils

#
Вездесущие реэкспорты - зло.
Они были придуманы для библиотек - чтобы скрыть сложную внутреннюю структуру, организовав единую точку импорта.
Если у вас похожая ситуация - есть сложный модуль, от возможных изменений в котором вы хотите оградить клиентский код - сделайте реэкспорт.
Но не надо реэкспортировать каждый файл приложения - это только вызывает вопросы у читателя - для чего же, думает он, так было сделано.
